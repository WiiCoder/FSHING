# 对话确认与文档同步规范

## 核心原则

FSHING 使用对话解决不确定性，使用文档保存已确认的事实和决定。

- Agent 必须主动把待确认项、冲突、风险和建议告诉用户，不能把问题只留在 Markdown 文件中等待用户发现。
- “确认需求”表示用户与 Agent 已解决会影响行为、范围或验收的关键问题；它不等于审批，也不要求 **approved_at**、签字或审批记录。
- 需求进入规划或实现阶段的依据是：当前 Spec 足够明确、没有未解决的阻塞决定，并且确认结果已经写回权威文档。
- 测试证据、发布门禁和正式发布事实仍然必须真实存在，不能用对话确认替代。

已有项目中的 **approved_at**、审批列或签字记录可以作为历史元数据保留，不需要为迁移本规则而删除；新流程不得要求填写、刷新或检查这些字段，也不得用它们阻断规划和实现。

## 主动确认循环

每个 fs 阶段都执行以下循环：

1. 读取用户请求、当前需求文档、相关发布记录和实际代码或测试状态。
2. 区分可通过仓库发现的事实与必须由用户决定的产品选择。先自行查证事实，不把可发现问题抛给用户。
3. 在对话中主动列出：当前理解、待确认项、每个选项的影响和推荐方案。一次优先处理一到三个最关键问题。
4. 仅暂停受阻塞问题影响的工作；不受影响的读取、分析和安全准备工作可以继续。
5. 用户确认后，在同一轮工作中立即更新权威文档和所有受影响的派生文档，然后再继续规划、实现、测试、评审或发布工作。
6. 如果不存在实质性待确认项，明确说明需求已足够清晰并继续执行用户已请求或明确授权的下一阶段，不增加审批步骤。

## 必须主动提出的问题

以下问题不得只写入文档：

- 目标用户、范围、非范围或核心流程存在多种合理解释；
- API、数据、权限、兼容性或关键验收条件不明确或互相冲突；
- 当前代码行为与 Spec 不一致，且无法从证据判断哪一方代表预期行为；
- 目标版本、组件归属、迁移、回滚或发布范围需要产品取舍；
- 测试发现验收标准缺失、不可验证或与实际需求不一致；
- 评审发现实现包含当前 Spec 未记录的行为；
- 任何假设会实质改变用户可见行为、安全边界、数据语义或发布风险。

## 回写范围

确认后按变化类型同步：

| 变化 | 必须更新的权威与派生文档 |
|---|---|
| 范围、行为、API、数据、权限、配置、FR/NFR 或 AC | **spec.md**、Spec 版本与 Change Log、**docs/features/index.yaml**；同步受影响的 design、tasks、test-plan、verification 和 release 记录 |
| 目标发布版本或发布范围 | **spec.md**、feature index、原版本延期记录和目标版本 requirements；必要时同步 release index |
| 仅技术实现事实，不改变产品行为 | **design.md**、**tasks.md**、**test-plan.md** 和相关证据；不为此虚增 Spec 版本 |
| 测试预期或覆盖变化 | 必要时先更新 Spec，再更新 test plan；将受影响的旧结果重置为 not-run 并记录原因 |
| 实现与需求冲突 | 暂停受影响实现，执行 requirement-change；完成确认和回写后再恢复 |
| 发布例外、移除或豁免 | 对话确认影响和处置，写入发布记录；若改变需求目标或状态，同时同步需求文档 |

不得只更新派生文档来绕过 Spec 变更，也不得确认后把回写留给用户。

## 对话输出要求

每个 fs 命令在结束或暂停等待确认时，都应直接告诉用户：

- 当前已确认的需求理解；
- 仍待确认的问题及其影响；
- 本轮已经回写的文件和版本变化；
- 因确认而失效或需要重做的任务、测试和证据；
- 下一步将继续执行的 fs 阶段或真实阻塞条件。
