# 完成定义与发布门禁

## 需求 Ready

需求进入 **ready** 前必须满足：

- ID、标题、类型、优先级、负责人和组件明确；
- 范围、非范围、FR/NFR 和 AC 完整；
- 依赖、风险和关键决策已记录；
- target_release 已填写或明确保持未排期；
- 当前 Spec 版本完成产品和必要技术评审；
- approved_at 与 Change Log 一致。

## 实现完成

实现完成表示代码任务完成，但不代表可以发布：

- 所有 TASK 有结果及组件归属；
- 代码与批准的 Spec 版本一致；
- API、数据、权限和配置变化已实现并记录；
- 每个代码变更都有组件名和 commit/PR 证据；
- 相关单元、静态检查和组件测试已通过；
- 设计和任务文档与实际实现一致；
- 没有通过未批准的行为假设填补需求空白。

实现完成后进入 **verifying**。

## Release Ready

需求进入 **release-ready** 前必须满足：

- 所有适用 AC 均有 passed 的 TC 和证据；
- P0/P1 缺陷为零，其他已知问题有明确处置；
- 数据迁移、权限、安全、性能和回滚按适用范围验证；
- verification.md 已记录环境、命令、结果和偏差；
- feature review 没有未解决的阻塞发现；
- target_release 与发布需求清单一致。

## Released

需求进入 **released** 前必须满足：

- 对应版本已实际正式发布；
- 发布版本状态为 released；
- released_in 已填写且与发布版本一致；
- repositories.yaml 已记录所有实际发布仓库的精确 revision；
- 发布测试报告和发布说明已完成；
- 部署或正式发布证据已记录。

因此：

~~~text
代码完成 ≠ 测试完成 ≠ release-ready ≠ released
~~~

## 不允许的替代证据

以下内容不能单独作为完成依据：

- 文件存在；
- AI 声称已经实现；
- 本地构建成功；
- 单个测试通过；
- PR 已创建或合并；
- 计划发布日期已到；
- 部署命令已执行但结果未确认。
